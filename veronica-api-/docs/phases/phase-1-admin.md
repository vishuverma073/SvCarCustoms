# Phase 1 (Backend) — Admin API

## What you'll build

The complete backend for the admin panel: auth, product/category/home/upload/settings CRUD, audit log, the new `home_config` singleton, plus stubs for orders that Phase 4 will fill in.

This is the biggest BE phase. **Plan for 3 sessions.** Before writing code, read [../admin-design.md](../admin-design.md) — it has the UX principles and schema additions you'll implement.

## Prerequisites

- [ ] Phase 0 complete — `@veronica/contracts@0.1.0` published, BE staging serves `/healthz` and `/categories` against real Postgres
- [ ] You've read [../admin-design.md](../admin-design.md)
- [ ] [TODO confirm with Ketan] Supabase Storage bucket `product-images` created with public read access
- [ ] [TODO confirm with Ketan] At least one admin user account to bootstrap (real email + a strong password)
- [ ] Razorpay account NOT needed yet (Phase 4)

## Success criteria

- Admin can log in via email + password from the FE staging build
- All admin CRUD endpoints work and reject non-admin tokens with 403
- Every admin mutation creates an `audit_log` entry
- `home_config` table exists, has a single row, is editable via `PUT /admin/home`
- Image upload returns a public URL hosted on Supabase Storage
- `@veronica/contracts@0.2.0` published with all admin schemas
- Integration milestone [M1](../integration-milestones.md#m1--after-phase-1-admin) passes when FE Phase 1 completes

## Estimated effort

3 sessions (~9 hours).

## Coordinate with FE

- The FE intern will be developing the admin UI against MSW mocks of *your* endpoint shapes. **Schemas in `@veronica/contracts` are the contract.** When you change a schema, ping the FE intern same day with the version + diff.
- FE will need the staging Supabase Storage URL added to their `next.config.ts` image domains. Tell them as soon as the bucket is created.
- The `home_config` shape (especially section keys) needs to match what FE renders on the storefront in Phase 2. Lock the section keys early: `hero`, `bestsellers`, `categories`, `new`, `featured`, `promo`.

## Task 1.3 — `requireAdmin` middleware with Redis cache

**Context**: Per agent council, re-check `is_admin` on every admin request (don't trust just the JWT) — but cache the DB lookup in Redis for 60s to avoid hitting DB on every request.

**Files to touch**:
- `apps/api/src/middleware/auth.ts` — add `requireAdmin`
- Tests

**Suggested Claude Code prompt**:
> Add `requireAdmin: MiddlewareHandler` to `apps/api/src/middleware/auth.ts`.
>
> Logic:
> 1. Read `Authorization: Bearer <token>` header
> 2. `verifyAdminAccess(token)` — on failure, return 401
> 3. Check Redis cache `admin:${userId}` (TTL 60s) for the user record
> 4. On cache miss: query `users` table for `id = sub`, must have `is_admin = true`. Cache result.
> 5. If `is_admin !== true`, return 403
> 6. `c.set("adminUserId", userId)` and `c.set("adminUser", userRecord)`
> 7. Call next()
>
> Note: skip Redis cache if `LOG_LEVEL=debug` to make local testing easier.
>
> Tests:
> - valid admin token → next called
> - customer token (no admin claim) → 401 (different issuer)
> - missing token → 401
> - revoked admin (is_admin set to false in DB) loses access within 60s → test by manually setting in DB, waiting for cache TTL
> - malformed token → 401

```bash
# Expect: 200

```

**Acceptance criteria**:

## Common pitfalls across this phase

- **Confusing admin vs customer JWT.** Different secrets, different issuer (`iss` claim). Don't accidentally use customer JWT validation on admin routes.
- **Storage bucket permissions**: must be public-read. If the URLs returned by upload work for you (the admin) but not for storefront visitors, the bucket is misconfigured.
- **bcrypt rounds**: keep 10 for Workers compatibility. If you bump it later, test login latency.
- **Schema migrations vs db:push**: use `pnpm db:generate` to create migration files, commit them. `db:push` is for fast iteration during dev — don't rely on it in CI/prod.
- **The audit log can grow fast**. Plan for archival/partitioning in Phase 6 if you cross 100k rows.
